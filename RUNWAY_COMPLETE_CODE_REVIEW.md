**Project:** `Projects/ACTIVE/apps-platforms/runway`  
**Scope:** Next.js 14 + Supabase + Python music discovery dashboard  
**Review focus:**
1. Exhaustive CLAUDE reference audit

4. Migration strategy for the CLAUDE persona
5. Security hardening checklist

## 0. Executive Summary

The codebase contains **two independent personas — KIMI and CLAUDE** — but there is **zero actual LLM integration**. The `agent` value is only a cosmetic label stamped onto tracks, playlists, and agent runs. The discovery pipeline is purely deterministic:

```
buildQuery(label + genre + energy + BPM)
  → searchSpotify() + searchTidal()
  → filterByReleaseDate()
  → deduplicateTracks()
  → save to DB
```

There are **no calls to `anthropic.com`, `claude.ai`, `openrouter.ai`, or any other LLM provider** anywhere in the runtime code.

**Most urgent issues:**
- `app/api/run-prompt/route.ts` creates playlists **without `user_id`**, but migration `008_playlist_ownership.sql` made `playlists.user_id NOT NULL`. This route will **HTTP 500 on every playlist creation**.
- `tracks` and `agent_runs` tables still have **no `user_id` column**, breaking the ownership model started in migration 008.
- Hardcoded credentials, bypassable middleware, and missing authorization checks are documented in `docs/security-audit.md` and `CRITICAL_PATH_FIXES.md`.

**Recommended path:**
1. Fix the `user_id` bugs first.
2. Add a real LLM layer via **OpenRouter** under `lib/llm/`.
3. Keep the **CLAUDE persona name** (Option A) to avoid a painful DB migration; just change what powers it.
4. Harden security in parallel.

## 2. Critical Bug Report

### 2.1 Confirmed: `app/api/run-prompt/route.ts` inserts playlists without `user_id`

**File:** `app/api/run-prompt/route.ts`  
**Lines:** 146–155

```ts
const { data: playlistRecord, error: playlistError } = await supabase
  .from('playlists')
  .insert({
    name: created.name,
    agent,
    service,
    external_id: created.id,
    track_count: created.track_count,
    prompt_name: promptName,
  })
  .select()
  .single()
```

**Problem:** The insert omits `user_id`.

**Migration 008 (`supabase/migrations/008_playlist_ownership.sql`) made `playlists.user_id NOT NULL`:**

```sql
ALTER TABLE playlists
  ADD COLUMN IF NOT EXISTS user_id uuid REFERENCES auth.users(id);

UPDATE playlists
SET user_id = (SELECT id FROM auth.users ORDER BY created_at ASC LIMIT 1)
WHERE user_id IS NULL;

ALTER TABLE playlists
  ALTER COLUMN user_id SET NOT NULL;
```

**Result:** Every `POST /api/run-prompt` that successfully creates a Spotify/Tidal playlist will **fail with HTTP 500** when Supabase rejects the insert:

```
new row for relation "playlists" violates check constraint "playlists_user_id_not_null"
```

**Fix:** Add `user_id: user.id` to the insert. This route already extracts `user` at line 54.

```ts
.insert({
  name: created.name,
  agent,
  service,
  external_id: created.id,
  track_count: created.track_count,
  prompt_name: promptName,
  user_id: user.id, // <-- FIX
})
```

### 2.2 Confirmed: `app/api/discover/route.ts` does NOT create playlists, but still writes un-owned rows

The prompt claimed `app/api/discover/route.ts` inserts playlists without `user_id`. That is **not accurate** — `discover/route.ts` only inserts into `agent_runs` and `tracks`:

### 2.3 Confirmed: `tracks` and `agent_runs` lack `user_id` columns

**Migration 008 only adds `user_id` to `playlists`.** It does not touch `tracks` or `agent_runs`.

`CRITICAL_PATH_FIXES.md` already recommends adding these columns:

```sql
ALTER TABLE playlists ADD COLUMN IF NOT EXISTS user_id uuid REFERENCES auth.users(id);
ALTER TABLE tracks ADD COLUMN IF NOT EXISTS user_id uuid REFERENCES auth.users(id);
ALTER TABLE agent_runs ADD COLUMN IF NOT EXISTS user_id uuid REFERENCES auth.users(id);
```

**Current state:**
- `playlists` has `user_id` (NOT NULL after migration 008).
- `tracks` has **no** `user_id`.
- `agent_runs` has **no** `user_id`.

This breaks the ownership model and leaves the RLS policies in migration 008 unable to scope `tracks` and `agent_runs` to users.

### 2.4 Proposed fixes

#### Fix A — Add `user_id` to `tracks` and `agent_runs`

Create `supabase/migrations/009_ownership_tracks_agent_runs.sql`:

```sql
-- Add user ownership to tracks and agent_runs
ALTER TABLE tracks
  ADD COLUMN IF NOT EXISTS user_id uuid REFERENCES auth.users(id);

ALTER TABLE agent_runs
  ADD COLUMN IF NOT EXISTS user_id uuid REFERENCES auth.users(id);

-- Backfill existing rows to the earliest-created auth user
UPDATE tracks
SET user_id = (SELECT id FROM auth.users ORDER BY created_at ASC LIMIT 1)
WHERE user_id IS NULL;

UPDATE agent_runs
SET user_id = (SELECT id FROM auth.users ORDER BY created_at ASC LIMIT 1)
WHERE user_id IS NULL;

-- Make NOT NULL
ALTER TABLE tracks
  ALTER COLUMN user_id SET NOT NULL;

ALTER TABLE agent_runs
  ALTER COLUMN user_id SET NOT NULL;

-- Indexes
CREATE INDEX IF NOT EXISTS idx_tracks_user_id ON tracks(user_id);
CREATE INDEX IF NOT EXISTS idx_agent_runs_user_id ON agent_runs(user_id);

-- Tighten RLS policies
DROP POLICY IF EXISTS "tracks_select" ON tracks;
CREATE POLICY "tracks_select" ON tracks
  FOR SELECT TO authenticated USING (user_id = auth.uid());

DROP POLICY IF EXISTS "tracks_insert" ON tracks;
CREATE POLICY "tracks_insert" ON tracks
  FOR INSERT TO authenticated WITH CHECK (user_id = auth.uid());

DROP POLICY IF EXISTS "agent_runs_select" ON agent_runs;
CREATE POLICY "agent_runs_select" ON agent_runs
  FOR SELECT TO authenticated USING (user_id = auth.uid());

DROP POLICY IF EXISTS "agent_runs_insert" ON agent_runs;
CREATE POLICY "agent_runs_insert" ON agent_runs
  FOR INSERT TO authenticated WITH CHECK (user_id = auth.uid());
```

## 3. LLM Integration Architecture

### 3.1 Core principle: keep personas, replace the engine

The product already distinguishes two taste profiles:

| Persona | Brand | Intended style |
|---------|-------|----------------|
| KIMI | Polished/mainstream | crowd-pleasing, high-production |
| CLAUDE | Underground/raw | deeper, lesser-known, leftfield |

These should become **two system prompts** routed through the same LLM client. The agent label is still stored in the DB; only the curation logic becomes intelligent.

### 3.2 Recommended provider: OpenRouter

**Why OpenRouter over native Kimi API:**
- One key, many models (`anthropic/claude-3.5-sonnet`, `openai/gpt-4o`, `moonshot-ai/kimi-k1.5`, `deepseek/deepseek-chat`, etc.).
- Lets the **CLAUDE persona actually run on a Claude model** — a fun product-fidelity win.
- Model strings can be swapped without provider migrations.
- Standard OpenAI-compatible API, so the client code is small.

**Fallback:** keep a `KIMI_API_KEY` env var option for a native Moonshot integration if needed.

### 3.3 Proposed `lib/llm/` module

```
lib/llm/
├── client.ts          # OpenRouter/Kimi HTTP client + retries + timeout
├── types.ts           # LLM request/response/persona types
├── prompts.ts         # System prompt templates for KIMI and CLAUDE
├── curation.ts        # Track scoring / re-ranking via LLM
├── query-enhancer.ts  # Convert prompt into smarter search queries
└── playlist-meta.ts   # Generate playlist name + description
```

```ts
  temperature?: number
  max_tokens?: number
  responseFormat?: { type: 'json_object' | 'json_schema'; schema?: object }
}

export async function callLLM(opts: LLMOptions): Promise<string> {
  const apiKey = process.env.OPENROUTER_API_KEY
  if (!apiKey) throw new Error('OPENROUTER_API_KEY is not set')

  const model = process.env.LLM_MODEL ?? DEFAULT_MODELS[opts.persona]
  const timeoutMs = parseInt(process.env.LLM_TIMEOUT_MS ?? '30000', 10)

  const abort = new AbortController()
  const timer = setTimeout(() => abort.abort(), timeoutMs)

  try {
    const res = await fetch(`${OPENROUTER_BASE}/chat/completions`, {
      method: 'POST',
      signal: abort.signal,
      headers: {
        Authorization: `Bearer ${apiKey}`,
        'Content-Type': 'application/json',
        'HTTP-Referer': process.env.NEXT_PUBLIC_SITE_URL ?? '',
        'X-Title': 'Runway Music Discovery',
      },
      body: JSON.stringify({
        model,
        messages: opts.messages,
        temperature: opts.temperature ?? 0.4,
        max_tokens: opts.max_tokens ?? 2048,
        ...(opts.responseFormat?.type === 'json_object'
          ? { response_format: { type: 'json_object' } }
          : {}),
      }),
    })

    if (!res.ok) {
      const body = await res.text().catch(() => '')
      throw new Error(`OpenRouter ${res.status}: ${body.slice(0, 200)}`)
    }

    const data = await res.json()
    return data.choices?.[0]?.message?.content ?? ''
  } finally {
    clearTimeout(timer)
  }
}
```

#### `lib/llm/curation.ts`

```ts
import { callLLM } from './client'
import { systemPrompt, curationUserPrompt } from './prompts'
import type { CurationInput, CuratedTrack, TrackCandidate } from './types'

export async function curateTracks(input: CurationInput): Promise<CuratedTrack[]> {
  const candidateText = input.candidates
    .map((t, i) => `${i + 1}. "${t.title}" by ${t.artist} (${t.source})`)
    .join('\n')

  const content = await callLLM({
    persona: input.persona,
    messages: [
      { role: 'system', content: systemPrompt(input.persona) },
      { role: 'user', content: curationUserPrompt(input.persona, input.promptName, candidateText, input.targetCount) },
    ],
    responseFormat: { type: 'json_object' },
  })

  const parsed = JSON.parse(content)
  const ranked: CuratedTrack[] = (Array.isArray(parsed) ? parsed : parsed.tracks ?? [])
    .map((r: any, idx: number) => ({
      ...input.candidates.find((c) => c.track_id === r.track_id) ?? input.candidates[idx],
      score: Number(r.score ?? 0),
      curation_reason: String(r.curation_reason ?? ''),
    }))
    .filter((t: CuratedTrack) => t.track_id)
    .sort((a: CuratedTrack, b: CuratedTrack) => b.score - a.score)

  return ranked.slice(0, input.targetCount)
}
```

```ts
// lib/rate-limit.ts
import { LRUCache } from 'lru-cache'

const cache = new LRUCache<string, number[]>({
  max: 1000,
  ttl: 60_000,
})

export function rateLimit(key: string, limit: number, windowMs = 60_000): boolean {
  const now = Date.now()
  const windowStart = now - windowMs
  const hits = (cache.get(key) ?? []).filter((t) => t > windowStart)
  hits.push(now)
  cache.set(key, hits)
  return hits.length <= limit
}
```

### 5.4 Circuit breaker for LLM failures

```ts
// lib/llm/circuit-breaker.ts
let failures = 0
let lastFailure = 0
const THRESHOLD = 5
const COOLDOWN_MS = 60_000

export function llmCircuitOpen(): boolean {
  if (failures < THRESHOLD) return false
  if (Date.now() - lastFailure > COOLDOWN_MS) {
    failures = 0
    return false
  }
  return true
}

export function recordLLMFailure() {
  failures++
  lastFailure = Date.now()
}
```

Use the circuit breaker in every LLM call. If open, fall back to deterministic logic.
